A planning & execution workflow for Claude Code

Plan like a spec.
Execute like a machine.

claude-planning-kit is a reusable starter kit you install into any project. From then on, Claude plans and ships improvements through a disciplined pipeline: codebase discovery, one batch of questions, a phased plan, an independent fact-checking review, and continuous conflict-safe execution — with every step persisted to disk, not chat history.

the loop  triage→(brainstorm)→discovery→questions→plan→independent review→your approval→continuous execution→final human checklist

01 How it works

Four ideas hold the whole thing together

The kit is two skills (feature-planning, plan-execution), two subagents (repo-explorer, plan-reviewer) and a 15-line triage router you paste into each repo's CLAUDE.md.

№ 1 — DISK OVER CHAT

Every phase writes a file

Each phase produces an artifact in planning/<slug>/ — snapshot, decisions, plan, review, execution logs — and commits it. Chat history is disposable; any session (or any teammate) resumes exactly where the last one stopped by reading the files.

№ 2 — RIGHT-SIZED PROCESS

Triage before anything

A tiny always-loaded router classifies every request. Trivial fixes skip the pipeline entirely; medium tasks get a mini-plan in chat; only genuinely risky work (schema, API contracts, auth, payments, multi-tenancy) pays the full-pipeline cost.

№ 3 — FRESH EYES ON EVERYTHING

Subagents with their own context

Discovery is delegated to read-only repo-explorer agents that return compressed, evidence-cited reports. The plan is judged by a plan-reviewer that starts with zero knowledge of the conversation — if the plan can't be validated from the artifacts alone, it's rejected as under-specified.

№ 4 — HUMANS AT THE EDGES

Your work is batched, never scattered

Everything agents need from you up front (keys, accounts, decisions) is collected once, as Phase 0. Everything only you can verify (manual QA, real payment tests, DNS, sign-offs) is sequenced after the last agent phase, into a single user-tasks.md checklist.

02 The workflow

Triage first, then the pipeline

Describe the improvement in plain language — Portuguese, English or Spanish; no need to say "plan". The router classifies it:

TierWhenWhat happens
DIRECT Trivial fix, ≤2 files, no schema / API / env / auth / payment impact Just does it. No ceremony.
LIGHT 3–10 files, moderate logic, still no contract impact A 5–10 line mini-plan in chat; you say "go", it executes in-session.
FULL New features, DB schema, API contracts, env vars, auth, payments, multi-tenancy The full pipeline below, run continuously in one session.

Brainstorm

optionalagent

Only when the idea is still vague. The brainstorming skill refines it into a design doc, which becomes the official feature request. Its handoff to Superpowers' own planner is deliberately blocked — this pipeline owns planning.

planning/<slug>/design.md

Discovery

subagents

Read-only repo-explorer agents investigate six areas — stack & boundaries, data layer, overlapping features & auth, integrations & config, deploy conventions, available tooling. Findings come back compressed, every claim cited with a file path. Nothing is guessed.

planning/<slug>/context-snapshot.md

Questions

you — pause nº 1

Every ambiguity the snapshot can't answer becomes a numbered question — with why it matters and a suggested default, so you can just reply "accept defaults". One batch, one pause. Your answers become the decision record.

planning/<slug>/decisions.md

Planning

agent

A two-page overview plus one self-contained file per phase, filled from literal templates: exact files to touch, pinned interfaces and signatures, ordered steps, acceptance criteria with a runnable verify command each, test commands, rollback note. Every phase is tagged executor: agent or executor: user — and user phases are pushed to the edges by rule.

plan.md  +  phases/phase-N.md  +  user-tasks.md

Independent review

subagent

The plan-reviewer reads only the artifacts — fresh context, zero conversation history — spot-checks 5–10 factual claims against the actual repository, and hunts for breaking contract changes, missing migrations, tenant-isolation holes, buried user steps, and undocumented divergence from the brainstorm's design doc.

planning/<slug>/plan-review.md
verdict: CHANGES REQUIRED → plan is fixed and re-reviewed, up to 3 rounds, until APPROVED — then remaining findings become your decisions

Approval & prerequisites

you — pause nº 2

You approve the reviewed plan and confirm the Phase 0 checklist: API keys, accounts, product decisions. Collected once, up front, and recorded on disk — no later session ever re-asks.

Continuous execution

agent

All agent phases run back-to-back in one session. Each phase is claimed via a local lock file plus an atomic claim ref pushed to the remote (parallel sessions and teammates never claim the same phase — even across machines), runs on its own branch or worktree, commits step by step, respects a scope fence, and must pass a strict definition of done — tests green, changelog updated, log written, branch merged.

planning/<slug>/execution/phase-N-log.md

Human checklist

you — the end

When only user phases remain, the kit hands you user-tasks.md: your manual tests, real-world validations and sign-offs, each with exact steps and what "pass" looks like — batched at the end so they never blocked the agents.

03 Get started

Three situations, three entry points

How you begin depends on whether the kit is already installed. In every case the day-to-day interface is the same: plain language in the chat — Portuguese, English or Spanish. Full install detail lives in INSTALL.md.

Situation A

Existing project,
kit not installed

You have a codebase and want it to adopt the workflow.

  1. Open Claude Code at the repository root and paste the entire contents of prompts/BOOTSTRAP-EXISTING-PROJECT.md. Nothing to edit — it already points at this repo.
  2. Claude installs everything itself: vendors the two skills and two subagents into .claude/, appends the triage router to CLAUDE.md, cherry-picks the four compatible Superpowers skills, verifies the result, and commits it on a chore/claude-planning-kit branch.
  3. Merge that branch. The project is wired — from the next message on, you're in Situation C.
› [paste BOOTSTRAP-EXISTING-PROJECT.md]
↳ installs · wires CLAUDE.md · verifies · commits — then stops; no feature work in that session
Situation B

New project,
from an empty folder

Nothing exists yet — not even git.

  1. Create an empty directory, open Claude Code inside it, and paste prompts/BOOTSTRAP-NEW-PROJECT.md — replacing only <PROJECT-NAME>.
  2. Claude initializes the repository (git, .gitignore, README), installs the kit, creates CLAUDE.md with the triage router, and scaffolds memory-bank/ — project brief, architecture, decisions, changelog, glossary — all marked TBD for the pipeline's discovery phase to fill in as real decisions are made.
  3. Describe your first feature. Planning starts at discovery; brand-new repos get the context scaffold instead of exploration.
› [paste BOOTSTRAP-NEW-PROJECT.md with PROJECT-NAME filled in]
↳ git init · kit installed · memory-bank/ scaffolded · committed — ready for feature nº 1
Situation C

Kit already installed —
how it triggers

There is no command to memorize. The router does the routing.

  1. Just describe the change. The triage block in CLAUDE.md is always loaded: it classifies every request and states the tier in one line. FULL-tier requests trigger the feature-planning skill on their own — you never have to say "plan".
  2. Vague idea? The pipeline runs the brainstorming skill first, turns it into a design doc, and continues into discovery from there.
  3. Executing and resuming: say "execute the plan", "continue the plan", "run the next phases" — or reference any planning/<slug>/ folder — and the plan-execution skill picks up from the artifacts on disk. Works in any session, days later, even from a teammate's machine.
› add an affiliate system with 10% commission
↳ triage: FULL — feature-planning starts discovery
› fix the typo on the checkout button
↳ triage: DIRECT — just fixes it, no pipeline
› continue the affiliate-system plan
↳ plan-execution resumes from planning/affiliate-system/ — locks, phases, definition of done

04 Scope, honestly

What it does — and deliberately doesn't

Most of the kit's value is in its refusals. The constraints below are enforced by the skills and the reviewer, not left to good intentions.

It does

enforced behaviors

  • Right-sizes the processTrivial requests are executed directly; only risky work pays for the full pipeline.
  • Persists everything to diskEvery phase writes an artifact to planning/<slug>/ and commits it; any session resumes from files alone.
  • Cites evidence for every claimDiscovery findings carry file paths; anything unverifiable becomes a question, never an assumption.
  • Fact-checks its own planAn independent reviewer validates the plan against the real codebase and the approved design doc before any code is written.
  • Executes continuously and in parallelLocal locks plus pushed claim refs arbitrate between simultaneous sessions — even on different machines; phases run on isolated branches with step-by-step commits.
  • Batches your work at the edgesPhase 0 prerequisites up front, one human checklist at the end — agents are never blocked mid-stream.
  • Composes with SuperpowersUses exactly four of its skills: brainstorming, test-driven-development, systematic-debugging, requesting-code-review.
  • Auto-updatesInstalled as a plugin marketplace; skill improvements propagate to every project on the next update.

It does not

hard refusals

  • Make product decisionsYou answer the questions, approve the plan, and own every trade-off. The kit structures decisions; it doesn't take them.
  • GuessNo unverifiable claim survives planning. If the repo can't prove it, it becomes a clarifying question.
  • Touch project files during planningDiscovery through review is strictly read-only. Code changes only begin after your approval.
  • Execute human workReal payment tests, DNS changes, third-party approvals, visual QA — it never claims a user phase; it sequences them for you.
  • Silently redesignTrivial drift (a renamed symbol, a moved file) is corrected and logged as a deviation without stopping; anything that would change a contract, schema, or acceptance criterion stops and asks — proposing the minimal correction.
  • Work directly on mainEvery phase runs on its own branch or worktree, and the integration branch is never left red.
  • Depend on chat historyNo /clear rituals, no "remember earlier". State lives on disk, sessions are replaceable.
  • Tolerate competing plannersSuperpowers' writing-plans, executing-plans, subagent-driven-development and using-git-worktrees are explicitly excluded — two planners fighting for the same trigger is worse than one.
